## weather+EOT
FEATURE = 天氣記錄 / 極端天氣 EOT (工期延誤申索). Two regimes (v58-weather-record.sql header):

A) LIVE WARNING BANNER (實時天氣, Part 1). Code: src/lib/hko.ts + src/components/WeatherBanner.tsx, rendered at ProjectDetail.tsx:314. Calls keyless HKO Open Data API directly (CORS-OK, no Supabase): warnsum + rhrread, lang=tc. Polls every 180000ms (3 min). fetchSiteWeather() returns SiteWeather { warnings: ActiveWarning[], stopWork: boolean, tempC: number|null, rainfallMm: number|null (max district past-hour), warningLines: string[], updatedAt: string|null }. ActiveWarning { code, name, label, severity, issueTime? }. severity = 'stopwork' (STOPWORK_CODES: TC8NE/SE/SW/NW, TC9, TC10, WRAINB=黑雨) | 'warning' (TC1/TC3/WRAINR=紅/WRAINA=黃/WMSGNL/WFNTSA/WL/WTS + any TC*) | 'info'. stopWork=true => red banner '惡劣天氣警告生效 — 應停止戶外高空/吊運工作'. Fails SILENTLY (returns null) on HKO error — purely supplementary. NOT module-gated (territory-wide public data) and NOT auth-dependent.

B) EXTREME-WEATHER RECORD + EOT CLAIM (Part 2). Two tables (v58):
- weather_events (TERRITORY-WIDE objective facts, NOT project-scoped, NOT module-gated): columns id uuid, hkt_date date, kind text CHECK in ('t8','t9','t10','black_rain','red_rain','amber_rain','rainfall_20mm','very_hot','cold','other'), station text (rain-gauge code for rainfall_20mm, null for territory warnings), evidence jsonb (warnsum: {code,name,type,issueTime,expireTime,source:'warnsum'}; rainfall: {mm,station,source:'daily_rainfall_csv'}), created_at. unique(hkt_date,kind,station). RLS weather_events_select USING(true) for any authenticated. Written ONLY by SERVICE ROLE — no client insert/update policy (= denied). WEATHER_KIND_ZH (types.ts:132): t8=八號風球, t9=九號風球, t10=十號風球, black_rain=黑色暴雨, red_rain=紅色暴雨, amber_rain=黃色暴雨, rainfall_20mm=24h雨量>20mm, very_hot=酷熱, cold=寒冷, other=其他. Member read via RPC get_recent_weather_events(p_days int default 120) [security definer, NOT module-gated].
- project_weather_claims (PER-PROJECT EOT rows, MODULE-GATED 'weather'): columns id uuid, project_id uuid (FK projects, on delete cascade), hkt_date date, trigger text NOT NULL (summary e.g. '黑雨 + 雨量 78mm'), on_critical_path boolean (受影響工序喺關鍵路徑?), ready_to_work boolean (天氣許可下本可施工?), tidy_days numeric>=0 (善後/清理需時 日), claim_days numeric>=0 (此日申請EOT日數), note text, recorded_by uuid (FK user_profiles, on delete set null), created_at, updated_at. unique(project_id,hkt_date). These carry CEDD PAH Appendix 7.4 (Inclement Weather Report Form) discretionary fields for a Government-GCC claim.

DATA SOURCE / SYNC: supabase/functions/weather-sync/index.ts — cron-driven (every ~30 min), service-role writes to weather_events. Secret-gated (WEATHER_SYNC_SECRET via ?secret= or x-sync-secret header; 403 otherwise). Source 1 = warnsum (in-force warnings NOW — HKO has no warning-history API, so frequent polling needed; captures T8+/黑/紅/黃雨/酷熱/寒冷 for hktToday()). Source 2 = HKO daily rainfall CSV per station (RAIN_STATIONS HKO,HKA,SHA,TWN,TU1,JKB), objective >20mm/24h ground; CSV lags ~1 month so this is BACKFILL, scans last 45 days. Upsert onConflict 'hkt_date,kind,station' ignoreDuplicates.

PER-ROLE CAPABILITIES (EOT claim — recording):
- admin: always can record/edit claims (WeatherRecord canManage line 30; SQL can_edit_project_progress).
- pm: can record/edit IF in projects.assigned_pm_ids OR approved membership role 'pm' (WeatherRecord:31-32).
- main_contractor (approved member): can record/edit (WeatherRecord:32).
- general_foreman (approved member): can record/edit per the CLIENT gate (WeatherRecord:32 lists 'general_foreman'). - subcontractor / subcontractor_worker / owner / safety_officer: read-only in the UI (no record button). safety_officer is NOT a recorder here.
- ALL approved members (any role incl. workers/owner) can VIEW claims (pwc_select = can_view_project) and view territory weather_events (always). When 'weather' module is OFF, members get 0 claim rows (RLS conjunct) AND the route/tile are hidden.

USER WORKFLOW (manager): open 項目 → 工具 tab → tap 天氣記錄 tile (CloudRain, '實時警告 · 極端天氣日 · EOT 延期申索') → page lists dates (union of weather_events days + existing claim days, newest first). For each extreme-weather day, tap '記錄此日' (or '編輯申索') → fill: 觸發原因 (prefilled from WEATHER_KIND_ZH of that day's events), 關鍵路徑 checkbox (default true), 本可施工 checkbox (default true), 善後日數, 申請EOT日數 (default '1'), 備註 → 儲存 (upsert onConflict project_id,hkt_date). WHO-SEES-WHAT (RLS): weather_events visible to every authenticated user territory-wide (ungated). project_weather_claims visible to project members via can_view_project AND project_module_enabled('weather'). Writes need can_edit_project_progress AND recorded_by=auth.uid() AND module enabled. Deletes NOT module-gated (admin can clean up after disabling). Live banner + AI get_weather_outlook are module-independent (public HKO).

3-5 COMMON CONFUSIONS: (1) The daily-log 天氣 dropdown (施工日誌) is a SEPARATE thing from 天氣記錄/EOT — easy to conflate. (2) Objective grounds (T8/黑雨/紅雨/24h>20mm, private SFBC/房署 = countable) vs Government-GCC (discretionary, needs 關鍵路徑/可施工 justification) — the page's footnote (WeatherRecord.tsx:107-109) states this but a tutorial must reinforce it. (3) Rainfall_20mm days appear LATE (CSV lags ~1 month) so a wet day may not show immediately — users expect real-time. (4) The empty state '暫時冇記錄到極端天氣日…天氣同步每30分鐘自動執行' is shown for BOTH 'genuinely no extreme weather' and 'weather module disabled / RLS returned 0' — misleading. (5) 申請EOT日數 ≠ calendar days lost: claim_days is what you claim per day (often 1, plus tidy_days for cleanup); the running total is informational, not an approved figure — the Engineer/業主 still adjudicates.

EXPORT FIELDS for a proper EOT report (currently only ad-hoc CSV at WeatherRecord.tsx:83-89): 日期(hkt_date), 觸發(trigger), 關鍵路徑(on_critical_path), 可施工(ready_to_work), 善後日數(tidy_days), 申請EOT日數(claim_days), 備註(note), + (recommended addition) objective evidence joined from weather_events for that hkt_date: kind (WEATHER_KIND_ZH), station, evidence.mm / evidence.code / evidence.issueTime / evidence.expireTime, + project name/number header, + total claim_days row.

## modules-gating-ripple
MODULE SYSTEM (v59) — authoritative facts for authoring a 教學 + module-audit export.

PER-ROLE CAPABILITIES:
- admin: ONLY role that can toggle modules. Reaches it via /admin -> per-project '模組' link (AdminProjects.tsx:174 -> /admin/projects/:id/modules). Can turn any of 12 non-core modules OFF/ON for a single project. Admins BYPASS every gate (ModuleGate.tsx:25, SiVoSwitcher showPtw, ToolsSwitcher showFiles/showEquipment) so they still see/manage a module they switched off.
- pm / main_contractor / subcontractor / subcontractor_worker / owner / safety_officer / general_foreman: CANNOT toggle modules. They only EXPERIENCE the effect — a disabled module's tab (ProjectDetail), 工具/簽核 card (ToolsSwitcher/SiVoSwitcher), sidebar link (Sidebar ProjectNavLinks) and route (ModuleGate) all disappear. RLS additionally returns 0 rows from the disabled feature table for them.

THE 13-MODULE CATALOGUE (src/lib/modules.ts MODULES, order is load-bearing): progress(進度, CORE/non-disableable, tabId), issues(問題, tab), si(工地指令, route 'si'), vo(變更指令, route 'vo'), ptw(工作許可證, route 'ptw'), weather(天氣記錄, route 'weather'), documents(文件, route 'files'), materials(物料, route 'materials'), contacts(聯絡人, route 'contacts'), timetable(行事曆, route 'timetable'), dailies(每日日誌, route 'daily'), equipment(機械/表格, route 'equipment'), assistant(助理, tabId — no route). 11 are RLS-gated feature tables; progress is core; assistant is also double-gated by ai_enabled.

STEP-BY-STEP ADMIN WORKFLOW (AdminProjectModules.tsx): 1) admin opens /admin, taps 模組 on a project row. 2) Page calls get_project_modules RPC, shows 13 rows each with label + mono key + a toggle (進度 shows a locked 核心 chip, cannot toggle). 3) Tapping a toggle calls set_project_module(p_project_id, p_module_key, p_enabled). 4) Toast 已啟用/已關閉「<label>」, list reloads. 5) Change propagates over realtime channel modules-${projectId} (ModulesContext.tsx:47) to every connected member — tabs/cards/links appear or vanish without refresh; if a member is sitting on a tab that gets disabled, ProjectDetail.tsx:125-133 bounces them to 進度.

WHO-SEES-WHAT (RLS visibility): default = absence-of-row => ENABLED (backwards-compat — live projects unchanged until an admin acts). DB project_module_enabled(project_id, key) coalesces missing row to true; client isModuleEnabled (ModulesContext.tsx:55) treats unknown/loading keys as enabled (fail-open). When disabled, the v59 RLS adds `and project_module_enabled(project_id,'<key>')` to SELECT and INSERT on the 11 feature tables, so non-admin members get 0 rows AND cannot insert. Admins are unaffected by the UI gates (and DB policies typically also admit admin).

3-5 COMMON CONFUSIONS: (1) 'I turned a module off but I (admin) still see it' — admins bypass all gates by design. (2) 'My open-issues count on the Dashboard dropped' — disabling 問題 zeroes that project's issues everywhere, including the cross-project Dashboard KPI (Dashboard.tsx:166), which cannot show a 'module off' marker. (3) 'I disabled 天氣記錄 but the weather banner still shows' — the live HKO banner (WeatherBanner) is separate always-on safety info, not the gated module. (4) 'A notification link opened the project home not the document' — a deep-link into a now-disabled module bounces via ModuleGate to /project/:id. (5) '進度 has no off switch' — it is the non-disableable core (modules.ts:40 core:true).

EXACT DATA SHAPE for a module-audit export (types.ts:1225-1238): table public.project_modules columns — project_id (uuid), module_key (text, one of the 13 ModuleKey values), enabled (boolean), updated_by (uuid -> user_profiles.id), updated_at (timestamptz). RPC get_project_modules() returns rows of ModuleState { module_key: string; enabled: boolean } (every catalogue key, override coalesced to true when absent). RPC set_project_module(p_project_id, p_module_key, p_enabled) is the only write path. Human labels come from MODULE_LABELS_ZH (modules.ts:55-69). No export UI exists today.

## signature+security
SLICE = signature + security (Update 3 sign-reauth/non-repudiation, plus the step-up 2FA + credentials supporting infra). Authoritative facts to author a 教學 + report export.

== PER-ROLE CAPABILITIES ==
- 二步驗證 (2FA/TOTP, SecuritySetup.tsx, /security-setup): EVERY logged-in role can enrol/disable their own TOTP factor (auth.mfa.enroll/unenroll). It is per-user, not per-project. App-to-app (Google/Microsoft Authenticator); works with synthetic phone.local emails. Once enrolled, step-up-gated actions (approve/reject SI·VO·PTW, membership/role change, document review/withdraw, progress delete, account delete, form_signoff) prompt for a 6-digit code (StepUpContext ACTION_CLASS_ZH lines 35-42).
- 簽名前確認身份 (sign re-auth, SignReauthContext): triggered automatically (no role distinction) at the moment of a PTW sign-off (PtwApproverBar.handleSign line 117) or a 法定表格 sign-off (EquipmentContext.signOff line 263) — ONLY when admin has flipped sign_reauth_enforced=TRUE (default FALSE). User re-enters their LOGIN PASSWORD; verified by verify-sign-password Edge fn which mints a 5-min sign_reauth_grants row (service-role only write).
- 合資格人士證明 (credentials, CredentialsSection.tsx, on Profile): ANY user uploads own credentials (addMyCredential, user_id=auth.uid()); verified_* stay null. admin / pm / safety_officer verify them via verify_user_credential RPC (step-up 'membership' gated). A verified+in-date credential whose credential_type matches form_templates.required_credential is what unlocks the 法定表格 簽署 button.
- 簽名證明 (本人 proof, SignatureProofCard): any project member who can_view_project can VIEW + EXPORT the certificate for any signoff in that project (get_signature_proof gate, v60 SQL lines 305-307).
- set_sign_reauth_enforced(bool): admin-only (v60 SQL lines 100-104). get_sign_reauth_enforced: any authenticated.

== STEP-BY-STEP WORKFLOW (sign re-auth, flag ON) ==
1. Approver opens PTW (status in_review) or equipment form instance. 2. Taps 簽署批准 / 簽署. 3. (step-up gate) if step_up_enforced ON and 2FA set, enters 6-digit code -> AAL2 -> mint_step_up_grant. 4. (for PTW) approve() advances chain; (for form) credential gate checks. 5. (sign-reauth gate) 簽名前確認身份 modal: re-enter login password -> verify-sign-password Edge fn -> sign_reauth_grants minted. 6. record_ptw_signoff / record_form_signoff runs assert_step_up THEN assert_sign_reauth THEN inserts the signoff. 7. 簽名證明 (本人) card appears on the detail page; tap 匯出簽名證明 (PDF) for the 勞工處 certificate.
FLAG OFF (current live default): steps 3 and 5 are skipped client-side (return true instantly, no prompt) and are no-ops server-side — signing is identical to pre-v60.

== WHO-SEES-WHAT (RLS) ==
- sign_reauth_grants: SELECT own only (v60 SQL lines 48-51); NO insert/update/delete policy -> only service role (verify-sign-password Edge fn) writes.
- step_up_grants: minted only post-AAL2 server-side.
- user_credentials: owner reads own; verify is admin/pm/safety_officer RPC.
- get_signature_proof: any can_view_project member of the signoff's project; non-member -> raise 你冇權查看呢個簽名證明.

== 4 COMMON CONFUSIONS ==
1. 'Why does it suddenly ask for my password when I sign?' — that is sign-reauth (proof of 本人 for 勞工處), distinct from the 2FA code; only appears after admin enables enforcement. 2. 'I entered my 2FA code, why ALSO a password?' — step-up (2FA, AAL2) and sign-reauth (fresh password) are TWO separate gates that can both be on. 3. 'I can't sign the 法定表格' — needs a VERIFIED, in-date 合資格人士證明 matching the template's required_credential; uploading isn't enough, admin/PM/safety_officer must verify. 4. '簽名證明 shows my role in English (subcontractor)' — current bug: raw DB role not ZH-mapped (SignatureProofCard line 154).

== EXACT DATA SHAPE for report export ==
get_signature_proof(p_kind text['ptw'|'form'], p_id uuid) returns jsonb (v60 SQL lines 328-354): signer{name,phone,role}, credential(jsonb snapshot: type, cert_no, valid_until — only for form), what_signed{kind,doc_id,doc_number,project_id,project, ptw_type?(PTW), template?+result?(form)}, signed_at, signature_present(bool), reauth{enforced,method:'password'}, tamper_evidence{table,ledger_seq,ledger_hash, integrity{intact,break_at,reason,count}}, attestation_zh.
Underlying columns: permit_signoffs(id, approval_id, ptw_id, signature_b64, created_at) -> approvals(actor_id) -> permits_to_work(number, ptw_type, project_id). form_signoffs(id, instance_id, project_id, result, payload, signed_by, signed_at, valid_until, signature_b64, credential_id, credential_snapshot) -> form_instances(template_id) -> form_templates(code, name_zh, required_credential, frequency_days). Signer identity: user_profiles.name + phone derived split_part(auth.users.email,'@',1) + project_members.role. Tamper anchor: audit_ledger(seq, hash, table_name, row_pk) + verify_integrity(0).
PDF export already implemented: exportSignatureProofPdf (export.ts line 1019) / SignatureProofInput (lines 999-1017).
