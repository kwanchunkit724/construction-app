## 系統現況（誠實版）

After this session's fixes the system is genuinely freeze-worthy on its strongest axis: the SI/VO/PTW approval ledger, the hash-chained audit_ledger, and additive-only migration discipline are mature and well-engineered, and the v64-v67 role/RLS widenings closed the loudest coherence gaps. But the dispute-survival spine — the product's one reason to exist — has a tight cluster of cheap, high-consequence integrity holes that all bake in permanently at freeze: account deletion still cascade-DELETES statutory 施工日報 (v11 dailies.user_id:19, explicitly left 'unchanged' by v20:45-46 on a rationale that wrongly lumps it with notification/membership tables); the issues UPDATE policy is USING-only with NO with check (v66:19-32, grants reporter_id=auth.uid()), so any reporter can self-resolve or mis-route their own issue with zero tamper evidence; and the audit_ledger watched array (v51:100-104) omits issues, issue_comments, dailies, AND materials — the four most litigable WhatsApp-replacement records sit entirely outside the hash chain. Compounding this, issue-photos is a PUBLIC uncompressed bucket (v4:126-127) leaking cross-project evidence and free-tier egress, and no storage bucket has a server-side size/MIME limit (zero matches across supabase/). None of these are features — they are integrity defects that are impossible (deleted dailies, un-hash-chained forged rows) or expensive (live public-bucket migration) to fix after a freeze. Net: ready to freeze only after a tight, almost-entirely-server-side integrity pass — and only if native ships first (see GO/NO-GO gate). Do NOT add features.

Closing the dailies cascade, the issues forgery window, and the four-table ledger gap means that for the whole freeze duration every core record is anonymize-on-delete, immutable-where-it-must-be, and tamper-evident — turning silent integrity failures into impossible or detectable ones with zero ongoing operator attention. ### T1.1 (S) Repoint dailies.user_id FK to user_profiles ON DELETE SET NULL + drop NOT NULL

**點解：** Account deletion (Apple-compliance-required, must keep working) currently cascade-DELETES every 施工日報 the user authored — irreversible loss of statutory dispute records. v20 fixed exactly this class for every other authored table but explicitly left dailies CASCADE 'unchanged', wrongly grouping a statutory record with notification/membership rows where cascade is correct. Once a daily is gone it cannot be reconstructed; this is the single worst freeze-baked defect. Reuse the in-repo _v20_repoint_fk helper.

**證據：** `supabase/v11-dailies-schema.sql:19 (user_id not null references auth.users on delete cascade); supabase/v20-delete-account-fk-cascade.sql:45-46 lists 'CASCADE-already columns (dailies.user_id ...) are unchanged'; _v20_repoint_fk defined at v20:53-66`

### T1.2 (M) Add BEFORE UPDATE column-guard trigger on issues (immutable reporter_id/reporter_role/project_id/created_at; resolved_by forced to auth.uid(); current_handler_role constrained to legal transitions)

**點解：** The issues UPDATE policy is USING-only with NO with check clause and includes reporter_id=auth.uid(), so any reporter down to subcontractor_worker can PATCH their own issue to resolved with resolved_by set to a PM, or jump the handler role to bypass the escalation chain — directly forging the dispute record. The documents register got the full guard treatment (v40-split/3); issues is the symmetric table that never did. This cannot be expressed in RLS because USING/WITH CHECK see only the NEW row, never OLD — it needs the proven BEFORE UPDATE trigger pattern.

**證據：** `supabase/v66-supervisor-write-rights.sql:19-32 (policy is using-only, includes reporter_id=auth.uid(), no with check); src/contexts/IssuesContext.tsx:171/192/213 (resolveIssue/reopenIssue write status/resolved_by via raw supabase.from('issues').update); grep confirms no 'before update on issues' guard exists anywhere in supabase/`

### T1.3 (S) Add issues, issue_comments, dailies, AND materials (plus drawings/drawing_versions) to the v51 audit_ledger watched-table array

**點解：** Even after the column-guard, a service-role/dashboard edit or an admin DELETE leaves no trace because these tables are outside the hash chain — yet they ARE the WhatsApp-replacement dispute artifacts. The critique correctly caught that the draft omitted dailies and materials: dailies is the very statutory record the FK-repoint protects (and it has a live delete policy at v11:96), and material orders are litigable ('ordered X on date Y, never came'). An admin issue DELETE also cascade-vaporizes the entire issue_comments thread (v4:35 on delete cascade) with no trace today. One-line additions to an existing idempotent DO loop; zero client impact; impossible to backfill historically once frozen. Execution-test that the AFTER DELETE on issues ledger row fires before the comment cascade.

**證據：** `supabase/v51-audit-ledger-tamper-evidence.sql:100-104 watched array omits issues/issue_comments/dailies/materials; supabase/v4-issues-schema.sql:95-96 admin DELETE on issues + :35 issue_comments on delete cascade; supabase/v11-dailies-schema.sql:96 dailies_delete policy`

**點解：** Size/MIME enforcement is currently client-JS-only, so a stale cached native build, a direct REST upload, or an SDK change bypasses every guard and can push arbitrary objects toward the 1GB cliff. This is the single most durable storage backstop — it survives a frozen/stale client untouched, which is exactly why it (not client compression) is what actually protects the freeze window. Trivial additive UPDATE storage.buckets, zero backward-compat risk.

**證據：** `grep for file_size_limit/allowed_mime_types across supabase/ returns ZERO matches; buckets created bare in v4-issues-schema.sql:126, plus v8 drawings, v9-si-vo, v40-split/5`

**證據：** `supabase/v4-issues-schema.sql:126-127 (public=true), :135-137 ('Public read issue photos', no auth/scope); src/contexts/IssuesContext.tsx:107 (stores photos array), :131-134 (upload + getPublicUrl); src/contexts/DrawingsContext.tsx:447-461 (createSignedUrl on private bucket, explicit 'NEVER getPublicUrl on this private bucket' pitfall comment)`

## ⛔ 唔好做（8）

- **Flip sign_reauth_enforced / step_up_enforced ON before the native re-auth UI ships** — Enforcement only fires inside SECURITY DEFINER RPCs and depends on a native client prompt that is in the PENDING rebuild. Flipping the flag without the UI would hard-break every signing path (PTW/動火證, statutory forms) for live iOS users — a direct constraint violation (must not break App Store users). The correct sequence is unchanged: ship native, THEN flip. See the GO/NO-GO gate in open_risks — this sequencing is now a freeze-timing gate, not a footnote.
- **Build hardware-bound device keypairs / biometric / ECDSA non-repudiation** — PLAN-CRITIQUE already established pgcrypto can't verify ECDSA and no biometric Capacitor plugin is installed — a multi-week native effort a freeze cannot justify. The shipped TOTP+password-reauth backend already gives ~90% of the value; enable what exists after native ships, don't chase the keypair design.
- **Add step-up / sign-reauth (MFA) to the issues write path as the forgery fix** — Those gates are flag-gated off and UI-dependent; issue resolution is high-frequency low-criticality field work. Wiring MFA in would add support burden and a native dependency. The durable fix is the cheap server-side BEFORE UPDATE column-guard, not another auth factor.
- **Add a retention/prune job to audit_ledger or worry about its growth on the 1GB tier** — Pruning breaks the hash chain (each row hashes the prior), destroying the tamper-evidence the feature exists for. The ledger is tiny next to drawings/permit photos; spend the storage worry on the server-side bucket limits and media compression, not the audit trail.
- **Try to fix the issues forgery with broader/column-level RLS policies** — Postgres RLS USING/WITH CHECK see only the NEW row, not OLD, so transition rules ('reporter may change X but not Y') cannot be expressed in policy SQL. Attempting it produces a brittle, partial guard. Keep the policy as the coarse row-touch gate and put the column logic in the BEFORE UPDATE trigger (the proven v40-split/3 / v55f mechanism).
- **Ship verify_audit_coverage as a standing RPC, or treat the schema dump as durable tooling** — Per the critique: for a no-iteration freeze, trigger coverage is fixed at apply time and the dump can never be re-validated without the same blocked DB access. Both are one-shot verification steps run during the integrity pass — adding them as permanent surface adds maintenance for zero freeze-window value. Run once, commit the dump as documentation, move on.
- **Add issue→progress-item FK, SI→VO forward badges, recurring-event support, or per-project-type default module profiles** — All are feature/polish work that expands surface area right before a freeze. The audit trail still functions via comments and free-text location; the back-link from VO→SI already exists; recurrence is disproportionate engineering. A freeze punishes added breadth — defer all of these.
- **Unify the auth.users-vs-user_profiles FK target split on the remaining v11 tables (contacts/events/materials)** — It's a cosmetic modeling asymmetry, not a data-loss or security risk, and unifying it means touching live-data FKs — exactly the kind of risky change a freeze should avoid. Document it as a known invariant. The dailies repoint in tier1 is the ONE exception, and only because it fixes an actual cascade-delete data-loss bug.

The whole pick ships in SQL + two client one-liners via the established Chrome SQL-editor apply channel, needs no native rebuild, is backwards-compatible (new-tables/additive-only, no destructive changes to progress_leaf_items/user_profiles), and adds near-zero maintenance burden — it fits every locked constraint.

**Over-engineering 警告：**
- verify_audit_coverage() admin RPC (tier1 #7 AND tier2): this is a meta-check that asserts every canonical table still carries trg_audit_ledger. For a FROZEN system the trigger set never changes after the freeze migration lands, so a coverage-drift detector guards against a threat (someone dropping a trigger) that cannot occur without a new migration — which by definition isn't happening during the freeze. The actual v51-typo class it cites is caught once, at apply time, by execution-verifying the freeze migration. Ship it as a one-time verification query during the integrity pass, not as a standing RPC that adds surface. Low harm but it's belt-on-belt for a no-iteration system.
- tier2 'freeze-time schema reconciliation baseline' (checked-in dump of information_schema + pg_constraint + pg_policies): valuable as a ONE-TIME artifact, but framed as ongoing. There is no schema_migrations ledger and MCP is blocked, so the dump can never be re-validated against the DB without the same blocked access — it's a write-once snapshot whose only consumer is a human doing manual diffs. Generate it, commit it, but don't dress it as durable tooling; it's documentation.
- tier3 pg_cron health-check pushing the admin (correctly self-demoted to optional/insurance, and the do_not_do reasoning is sound): a new monitored subsystem with its own failure mode is exactly the wrong thing to add right before a no-maintenance freeze. The recommendation already ranks it low — endorse keeping it OUT of the pre-freeze pick.
- tier3 storage-usage gauge: once tier1 lands server-side file_size_limit + compression, the 1GB cliff is structurally pushed out ~10x. A read-only admin gauge is genuinely useful but it is a new page to maintain for a number nobody will look at during a freeze; the server-side limit is the durable backstop and the gauge is redundant comfort.

- ISSUE-PHOTOS PRIVACY MIGRATION IS RISKIER THAN 'M': verified issues.photos stores FULL public URLs (IssuesContext.tsx:107 stores the array, :134 returns getPublicUrl). - FROZEN-WINDOW SIGNATURE WEAKNESS IS STRUCTURAL, NOT JUST A RISK: open_risk #6 correctly notes every PTW/動火證/statutory signoff captured before native ships is permanently the weaker session-backed kind. ### Audit-Trail Integrity & Non-Repudiation (`audit-integrity`)

The audit trail is genuinely strong and is the most carefully engineered part of this system.
